## 1. 비전

### 한 줄

> 모든 사람과 기업이 수십 개의 에이전트를 운영하는 시대에, 클로드코드 중심 CLI 기반 AI 워크스테이션을 만든다.

### 핵심 전략: 싸우지 않고 이긴다

기존 메신저/서비스의 **상위 집합**으로 올라간다.

```
Web 2.0 메신저: {사람 ↔ 사람}

ClawTalk:
  {사람 ↔ 사람}                    ← 기존 포함
  ∪ {사람 ↔ 에이전트}              ← 신규
  ∪ {에이전트 ↔ 에이전트}          ← 신규
  ∪ {에이전트 상태/권한/검증}       ← 신규

∴ ClawTalk ⊃ 카카오톡/텔레그램
```

AI 시대가 가속할수록 에이전트 축이 폭발. 기존 메신저 영역은 상대적 축소. **시간이 우리 편.**

### bhOS가 제품, 나머지는 모듈

```
bhOS (최종 제품)
├── ClawTalk     소통     에이전트 네이티브 메신저 (큰 시장)
├── ClawMemory   기억     관계 + 지식 + AI 메모리 (차별화 엔진)
├── ClawMeeting  미팅     녹음 + 전사 + 요약 (즉시 수익)
├── ClawNote     노트 UI  초기 옵시디언 → 자체 개발
└── NanoClaw     런타임   에이전트 실행 엔진 (오픈소스 기반)
```

### 장기 비전

```
Agent Economy (거래/결제) → Agent App Store (마켓플레이스)
→ ClawTalk + ClawMeeting + ClawMemory (제품군)
→ Claw Core (인증 + 관계 + 이벤트 버스 + Agent ID)
→ NanoClaw (에이전트 런타임)
→ bhOS (AI 워크스테이션)
```

### Claw Core (공유 계층)

모든 모듈이 의존하는 최소 공통 계층. 자동 설치.

| 구성 요소 | 역할 |
|----------|------|
| 인증 | Google OAuth / 자체 계정 |
| 사용자 프로필 | 기본 정보 |
| 관계 그래프 | Person + 관계 유형 + 온도 |
| 이벤트 버스 | 모듈 간 느슨한 연결 |
| Agent Identity | 에이전트 신원/신뢰 (향후) |

```prisma
model User {
  id    String @id
  name  String
  email String @unique
}

model Person {
  id           String @id
  name         String
  relationship String  // family, friend, colleague, client...
  temperature  Int     // 1-5
  ownerId      String
}

model Event {
  id        String   @id
  type      String   // "meeting.completed", "memory.updated"
  source    String   // "clawtalk", "clawmeeting", "clawmemory"
  payload   Json
  createdAt DateTime
}
```

### 이벤트 버스

모듈 A는 모듈 B의 존재를 모른다. 이벤트만 발행.

```
ClawMeeting: "meeting.completed" 발행
  → ClawMemory 있으면: 전사본 저장, 관계 업데이트
  → ClawMemory 없으면: 아무 일 없음 (에러 아님)

ClawTalk: "calendar.meeting_soon" 발행
  → ClawMemory 있으면: 브리핑 자료 생성
  → ClawMemory 없으면: 단순 알림만
```

구현: MVP는 SQLite 테이블 + 폴링. 확장 시 Redis pub/sub.

### 모듈별 조합

| 사용자 유형 | 설치 | 경험 |
|------------|------|------|
| 미팅 녹음만 | Core + Meeting | 가입→녹음→전사 |
| 에이전트 메신저 | Core + Talk | 에이전트 약속 잡기 |
| 세컨브레인 | Core + Memory + 옵시디언 | 관계 + 지식 |
| 풀 워크스테이션 | Core + 전부 | bhOS 완전체 |

### Federated Memory

파일은 제자리에 두고, 인덱스만 통합.

```
물리적 저장소 (변경 없음):

  ~/.claude/           Claude Code 메모리
  ~/nanoclaw/groups/   NanoClaw 에이전트 메모리
  ~/.claw/note/        ClawNote 마크다운

통합 인덱스 (신규):
  ~/.claw/index/
  ├── graph.db     관계 + 엔티티
  ├── vectors.db   임베딩
  ├── fts.db       전문 검색
  └── sources.json 인덱싱 대상
```

### 두 트랙: 로컬 + 클라우드

| | 로컬 (개발자) | 클라우드 (일반) |

같은 코어 엔진, 배포 모델만 다름.

## 3. 제품별 상세

### ClawTalk — 에이전트 네이티브 메신저

**역할**: 사람과 에이전트가 1:1, 1:다, 다:다로 소통하는 메신저
**시장**: 글로벌 메시징 $100B+ (장기)
**상세 PRD**: 이 문서의 [5장](#5-clawtalk-prd) 참조

핵심 기능:
- Room (사람 N + 에이전트 M 혼합 대화방)
- Task Engine (A2A 기반 상태 머신 + HITL)
- Agent Card (에이전트 등록/발견)
- Telegram Bridge (첫 진입점 → 웹앱 유도)
- Calendar + Time Slot Manager
- 온보딩 (가상봇 체험 → 30초 아하 모먼트)
- BYOK + OpenRouter (무료 체험)

---

### ClawMemory — 기억 + 관계 + 맥락

**역할**: 노트가 아닌 **기억 시스템**. 사람, 관계, 대화, 맥락을 통합.
**시장**: 다른 제품의 가치를 10배 증폭하는 차별화 엔진

#### 3개 계층

| 계층 | 내용 |
|------|------|
| **Knowledge** | 마크다운 노트 + 자동 온톨로지 추출(LLM) + 벡터 검색 |
| **Relationship** | 사람 엔티티 + 관계 유형(가족/친구/동료/거래처/고객) + 관계 온도 + 상호작용 히스토리 |
| **Context** | AI 에이전트 메모리 통합 + 세션 히스토리 + 시간축 |

#### 온톨로지 전략

> "마크다운 퍼스트 + 사용자 명시 관계 정의 + 자동 온톨로지 보강 + 벡터 검색"

- 사용자: 마크다운 작성 + 사람/관계 명시 등록
- 시스템: 자동 엔티티 추출 + 임베딩 + 온톨로지 보강
- 온톨로지를 사용자에게 강제하지 않음

#### 킬러 유즈케이스: 미팅 준비 자동화

```
캘린더 미팅 감지 → 참석자 관계 조회 → 과거 맥락 검색
→ 브리핑 자동 생성 → ClawTalk 전달 (1시간 전)
→ 미팅 녹음 (ClawMeeting) → 회의록 → 관계 업데이트 → 온도 조정
```

#### 옵시디언 전환 전략

```
Phase 0: 옵시디언 = UI, CLI = 관계/인덱싱
Phase 1: 옵시디언 플러그인 (관계 시각화, 클라우드 동기화=유료)
Phase 2: ClawNote 자체 클라이언트 (옵시디언 데이터 호환)
```

### ClawMeeting — 미팅 녹음 + 전사 + 요약

**역할**: 한국어 특화 미팅 전사 서비스
**시장**: 미팅 전사 (Fireflies $100M+ ARR, 검증됨)
**전략**: 한국어 끝판왕 품질, Day 1 클라우드 서비스, 즉시 수익

핵심 기능:
- 실시간 음성 녹음 + 한국어 전사
- 화자 분리 (diarization)
- 실시간 문단 정리 (Tiro 스타일)
- AI 요약 + 액션 아이템 추출
- ClawMemory 연동 (전사본 → 관계 업데이트)
- 클라우드 저장 (유료)

---

### ClawNote — 마크다운 노트 UI

**역할**: ClawMemory의 프론트엔드
**전략**: 초기 옵시디언으로 대체, 점진적 자체 개발

초기: 옵시디언 볼트가 데이터 저장소
후기: 자체 에디터 (옵시디언 마크다운 호환 유지)
수익: 클라우드 동기화 (옵시디언 Sync 모델, 플러그인으로도 제공 가능)

---

### NanoClaw — 에이전트 런타임

**역할**: 모든 모듈의 기반. 에이전트가 실제 작업을 수행하는 엔진.
**전략**: 오픈소스. 직접 수익 없음. 생태계 성장 동력.

이미 운영 중:
- Docker 컨테이너 격리
- MCP (도구 접근)
- 멀티채널 (Telegram, Slack, Discord, WhatsApp)
- BYOK (Claude OAuth, API Key, Ollama)

## 5. ClawTalk PRD

> 에이전트 네이티브 메신저 | v0.1 | 2026-03-20

### 5.1 제품 개요

**한 줄 정의**: 사람과 AI 에이전트가 함께 대화하고, 에이전트끼리 조율하고, 사람이 검증하고 승인하는 메신저.

**한 줄 피치**: "카카오톡이 사람 간 대화를 위한 메신저라면, ClawTalk은 사람과 AI 에이전트가 함께 일하는 메신저입니다."

**포지셔닝**: 기존 메신저와 경쟁하지 않는다. 에이전트 통신 계층을 올린다. HTTP가 TCP/IP를 대체하지 않았듯, ClawTalk은 Telegram/카카오톡 위에서 작동하면서 자체 클라이언트로 최적 경험을 제공한다.

**핵심 차별점 3가지**:
1. **에이전트 투명성** — 에이전트가 지금 뭘 하는지 실시간으로 보임
2. **에이전트 간 통신** — 에이전트끼리 자동 조율, 사람은 결과만 승인
3. **HITL 네이티브** — 승인/거부/위임이 메시지 프리미티브에 내장

**프로토콜 기반** — 내부적으로 ACP(Agent Communication Protocol) 사용:
- A2A(Google): 에이전트 간 Task lifecycle, Agent Card
- MCP(Anthropic): 에이전트의 도구 접근 (캘린더, API 등)
- Matrix: Room 모델, Federation, E2EE (향후)
- 신규: HITL 프리미티브, Multi-Agent Negotiation, Observability

**"제품 먼저, 프로토콜은 추출"** — ClawTalk이 성공하면 내부 ACP를 오픈 스펙으로 공개.

### 5.2 타깃 사용자

**1차 타깃: AI 파워유저 개발자**
- 에이전트를 이미 사용하는 개발자 (NanoClaw, CrewAI, Claude Code 사용자)
- Pain point: "내 에이전트가 뭘 하는지 모르겠고, 여러 에이전트를 관리하기 어려움"
- Developer-first GTM (Stripe/Vercel/Supabase 모델)

**2차 타깃: 얼리 어답터 비개발자**
- AI에 관심 많은 PM, 마케터, 운영팀
- Pain point: "에이전트에게 일을 시키고 싶은데 기술적 장벽이 높음"

**3차 타깃: 엔터프라이즈 팀**
- 에이전트를 업무에 도입하려는 조직
- Pain point: "보안, 감사 추적, 권한 관리가 필요"

### 5.3 핵심 기능 (MVP)

#### Room (대화방)

사람과 에이전트가 함께 참여하는 방.

- **참여자**: 사람 N명 + 에이전트 M개 (혼합)
- **메시지 타입**:
  - 텍스트/미디어 (일반 대화)
  - **Action** (에이전트 작업 요청)
  - **ApprovalRequest** (HITL 승인 — 버튼/폼)
  - **AgentEvent** (에이전트 간 통신 로그 — 투명성)
- **에이전트 상태 표시**: 각 에이전트 옆에 실시간 상태 배지
  - 대기 | 작업 중 | 승인 대기 | 완료 | 실패

#### Task Engine (작업 관리)

A2A Task lifecycle 기반 상태 머신.

- **상태**: SUBMITTED → WORKING → INPUT_REQUIRED → COMPLETED / FAILED
- **투명성**: 각 단계가 Room에 AgentEvent로 표시
- **HITL**: INPUT_REQUIRED 상태에서 사용자에게 승인 UI 제공
  - 복수 선택 토글 + 확인 버튼
  - 순차 승인: A가 먼저 선택 → B는 A의 선택 중에서 (PoC 검증 완료)
- **타임아웃**: 설정 가능한 자동 만료

#### Agent Card (에이전트 등록)

- 사용자가 자신의 에이전트를 등록 (이름, 설명, 기능 스키마)
- Room에 에이전트를 초대하면 자동 참여
- MVP: **Schedule Agent** 1종 기본 제공
- 향후: 에이전트 마켓플레이스

#### Telegram Bridge

- **역할**: 첫 접점 (Telegram) → 웹앱으로 점진적 유도
- Telegram 그룹 ↔ ClawTalk Room 양방향 동기화
- 인라인 키보드로 HITL 승인 (Telegram에서도 가능)
- `/schedule` 명령어로 약속 잡기

#### Calendar + Time Slot Manager

**캘린더 연동:**
- MVP: Google Calendar (OAuth2)
- 향후: Apple (CalDAV), Outlook (Graph API)
- MCP 서버로 추상화 — 에이전트는 `find_free_time(userId)` 하나만 호출

**Time Slot Manager (Calendly 스타일):**
```
사용자의 "진짜 가용 시간" =
  캘린더에 일정 없음
  ∩ 수동 잠금 아님 ("금요일 저녁은 가족 시간")
  ∩ 선호 시간대 내 ("저녁 약속은 18~21시")
  ∩ 기본 규칙 충족 ("주중 업무시간 제외")
```

- **기본값 (온보딩)**: 평일 18~21시, 주말 12~21시 오픈
- **커스터마이즈**: 반복 잠금, 일회성 잠금, 선호 시간대, 최소 간격

#### 온보딩 튜토리얼

```
Step 1: "Google 캘린더를 연동하세요" → OAuth 클릭 (10초)
Step 2: "가상 친구 '봇B'와 저녁 약속을 잡아보세요" → 자동 진행
  - 에이전트가 캘린더 확인 → 가상 친구와 협상 → 시간 제시
  - 사용자가 시간 선택 → "약속 확정!"
  - → "이것이 ClawTalk입니다. 에이전트가 알아서 조율하고, 당신은 승인만."
Step 3: "이제 진짜 친구를 초대해보세요" → 초대 링크 생성
```

#### BYOK + OpenRouter (AI 모델 연동)

- **체험**: OpenRouter 무료 크레딧으로 가입 즉시 사용 (설정 0)
- **본격 사용**: 자신의 API 키 연동
  - Claude (Anthropic API Key 또는 OAuth Token)
  - OpenAI
  - 로컬 LLM (Ollama)
  - OpenRouter (통합 게이트웨이)

## 6. 핵심 결정 기록 (ADR)

> 왜 이렇게 했는가. 미래의 나를 위한 기록.

### ADR-001: 기존 메신저와 "경쟁"이 아닌 "계층 추가" 포지셔닝

- **날짜**: 2026-03-19
- **결정**: 카카오톡/텔레그램을 대체하지 않고, 그 위에 에이전트 통신 계층을 올린다
- **이유**: 메신저의 네트워크 이펙트는 극복 불가. HTTP가 TCP/IP를 대체하지 않았듯, 새 패러다임은 기존 인프라 위에 올라간다.
- **대안**: 완전히 새 메신저 (네트워크 이펙트 장벽), 기존 메신저 플러그인 (기능 제한)

### ADR-002: "제품 먼저, 프로토콜은 추출" 전략

- **날짜**: 2026-03-19
- **결정**: 킬러 제품을 먼저 만들고, 내부 통신 규약을 나중에 오픈 스펙(ACP)으로 추출
- **이유**: Matrix(12년, 메인스트림 실패), ActivityPub(Mastodon 1000만 MAU) 등 프로토콜 퍼스트 전략의 한계. MCP는 Claude Code(제품)에서 출발해 프로토콜로 확산됨 — 이 경로를 따른다.
- **대안**: 프로토콜 퍼스트 (채택 느림), 기존 프로토콜 위에 빌드 (제약 많음)

### ADR-003: Developer-first GTM, "에이전트 투명성"이 wedge

- **날짜**: 2026-03-19
- **결정**: 개발자/AI 파워유저를 첫 타깃으로, "내 에이전트가 뭘 하는지 보이게 하는 것"이 핵심 가치 제안
- **이유**: 개발자가 에이전트를 가장 많이 쓰고, 투명성/HITL 니즈가 가장 강함. Stripe/Vercel/Supabase의 developer-first GTM이 증명된 전략.
- **대안**: 일반 사용자 대상 킬러 UX (activation barrier 높음), B2B 엔터프라이즈 (긴 영업 주기)

### ADR-004: PoC는 NanoClaw 통합이 아닌 독립 스크립트

- **날짜**: 2026-03-20
- **결정**: "약속 잡기" PoC를 독립 스크립트로 구현
- **이유**: NanoClaw 코드를 수정하면 기존 기능이 깨질 위험. 독립 스크립트로 빠르게 가설 검증 후, 검증된 패턴만 통합.
- **대안**: NanoClaw IPC 확장 (코드 복잡도 증가), Telegram Mini App (과도한 복잡도)

### ADR-005: 에이전트 간 통신은 공유 백엔드 경유

- **날짜**: 2026-03-20
- **결정**: Telegram 봇 간 통신을 공유 백엔드(in-memory state machine)로 구현
- **이유**: Telegram은 봇 간 메시지 수신을 원천 차단 (공식 정책). Slack/Discord도 채널 경유만 가능. 어떤 메신저에서든 에이전트 간 직접 통신은 불가능하므로, 별도 통신 계층이 필수. 이것이 ACP의 존재 이유 자체.

### ADR-006: 순차 선택 HITL 패턴

- **날짜**: 2026-03-20
- **결정**: 시간 선택 시 A가 먼저 복수 선택 → B가 A의 선택지 중에서 선택
- **이유**: 병렬 선택 시 양쪽이 다른 시간을 고르면 교착 상태. 순차 방식은 합의 확률이 높음. PoC 테스트에서 자연스러운 플로우 확인.
- **대안**: 병렬 선택 + 교집합 (불일치 시 재투표 필요), 에이전트 자동 결정 (HITL 부재)

### ADR-007: 캘린더 통합은 MCP 서버 추상화 계층으로

- **날짜**: 2026-03-20
- **결정**: Google/Apple/Outlook 캘린더를 MCP 서버로 추상화하여 에이전트는 단일 인터페이스로 접근
- **이유**: 사용자마다 다른 캘린더 사용 (Google 60%, Apple 25%, Outlook 15%). 에이전트는 "빈 시간 알려줘"만 요청하고, MCP 서버가 라우팅.
- **대안**: 각 캘린더 API를 에이전트가 직접 호출 (복잡도 폭발), 자체 캘린더 서비스 (마이그레이션 문제)

### ADR-008: bhOS가 OS 계층

- **날짜**: 2026-03-20
- **결정**: ~/bhOS/에 이미 프로젝트 관리, 컨텍스트, 세션 프로토콜이 존재. ClawTalk/ClawMemory/ClawMeeting은 bhOS 위의 모듈.
- **이유**: bhOS는 폴더 구조 + CLAUDE.md + NOW.md + 런북으로 이미 "개인 운영체제" 역할을 함. 중복 설계 불필요.

### ADR-009: Federated Memory

- **날짜**: 2026-03-20
- **결정**: 기존 파일을 옮기지 않고, ~/.claw/index/에 통합 인덱스(graph.db + vectors.db)만 구축.
- **이유**: 파일 이동 시 기존 워크플로우 파괴. 락인 방지(Claw 없이도 파일 직접 접근 가능). 마크다운 + SQLite 표준 형식 고수.

### ADR-010: 모듈 독립성 — Claw Core + 이벤트 버스

- **날짜**: 2026-03-20
- **결정**: Claw Core(인증 + 관계 그래프 + 이벤트 버스)를 공유 계층으로, 각 모듈은 독립 설치/실행 가능.
- **이유**: 사용자가 ClawMeeting만, 또는 ClawTalk만 골라 쓸 수 있어야 함.

### ADR-011: ClawMemory는 노트가 아닌 기억 시스템

- **날짜**: 2026-03-20
- **결정**: 3계층(Knowledge + Relationship + Context). 관계 유형, 관계 온도, AI 메모리 통합.
- **이유**: "노트 앱"이 아니라 "디지털 자아의 기억". 관계 그래프가 있어야 미팅 브리핑 자동화 같은 킬러 기능 가능.
- **온톨로지 전략**: 마크다운 퍼스트 + 사용자 명시 관계 + 자동 온톨로지 보강 + 벡터 검색

### ADR-012: 온보딩 = 30초 아하 모먼트

- **날짜**: 2026-03-20
- **결정**: 가입 → Google 캘린더 연동 → 가상봇과 약속 잡기 체험. Telegram이 첫 진입점, 웹앱으로 점진적 유도.
- **이유**: 카카오톡 성공 = "설치→친구추가→대화" 10초. 에이전트 가치를 체험하기도 전에 이탈하면 안 됨.

### ADR-013: 두 트랙 — 로컬(개발자) + 클라우드(일반)

- **날짜**: 2026-03-20
- **결정**: 같은 코어 엔진, 배포 모델만 다름. 개발자 = 맥미니 로컬 + BYOK + CLI. 일반 = 클라우드 가입 + 구독 + 스토리지 과금.
- **이유**: 로컬 퍼스트는 개발자 신뢰 확보. 클라우드는 일반 사용자 진입장벽 제거 + 수익.

### ADR-014: ClawMeeting = 한국어 특화, Day 1 클라우드, 즉시 수익

- **날짜**: 2026-03-20
- **결정**: Fireflies/Otter 계열의 미팅 녹음/전사 서비스를 한국어 끝판왕 품질로. 처음부터 클라우드.
- **이유**: 검증된 시장(Fireflies $100M+ ARR). 클라우드 인프라를 여기서 구축하면 다른 모듈 유료화 경로도 열림.

### ADR-015: 문서 역할 분리

- **날짜**: 2026-03-20
- **결정**: nanoclaw = "왜" + 전체 그림 (비전, 아키텍처, BM, 리서치). clawtalk = "뭘" + "어떻게" (PRD, Sprint, API 설계).
- **이유**: 하나의 문서에 전략과 실행이 섞이면 날카로움이 사라짐. 각 레포가 자기 문서를 소유.

## 7. PoC 검증 결과

> PoC 001: 에이전트 간 약속 잡기 데모 | 2026-03-20 | 상태: 검증 완료

### 가설

> "두 사람의 에이전트가 백엔드에서 자동으로 일정을 조율하고, 사람은 최종 승인만 하는 UX는 기존 메신저에서 불가능하며, 별도 통신 계층(ACP)이 필요하다."

### 아키텍처

```
Telegram 그룹 (UI)
├── 나(A) — 인간
├── 친구(B) — 인간
├── BotA (@Nano_Samanda_bot) — A의 에이전트
└── BotB (@BHtalk_friend_agent_bot) — B의 에이전트

공유 백엔드 (Node.js 단일 프로세스)
├── Grammy BotA instance
├── Grammy BotB instance
├── Coordinator (상태 머신)
└── Google Calendar API 연동
```

### 검증 결과

- [x] 봇 2개 동시 동작 (Telegram)
- [x] 에이전트 간 통신 (공유 백엔드 Coordinator)
- [x] 투명성 로그 ([에이전트 간 통신])
- [x] 복수 선택 HITL (토글 + 확인)
- [x] 순차 승인 (A 먼저 → B는 A 선택 중에서)
- [x] Google Calendar 실제 연동 (FreeBusy + Event 생성)
- [x] 솔로 모드 (가상 친구) 동작

### 핵심 발견

1. **에이전트 간 통신은 반드시 별도 계층 필요** — Telegram 내부로는 절대 불가. 이것이 ACP 존재 이유.
2. **투명성(observability) UX가 핵심** — 에이전트 간 통신 로그가 그룹에 표시되면 사용자가 "뭐가 일어나고 있는지" 바로 이해함.
3. **순차 선택이 병렬 선택보다 나음** — A가 먼저 복수 선택 → B가 그 중에서 선택하는 플로우가 자연스러움.
4. **인라인 키보드가 HITL의 핵심 프리미티브** — 토글+확인 패턴으로 복수 선택 가능.
5. **공유 백엔드 = Coordinator 패턴** — 상태 머신이 에이전트 간 조율의 중심. ACP 서버의 프로토타입.

### 해결된 기술 이슈

| 이슈 | 해결 |
|------|------|
| Telegram 봇 간 메시지 불가 | 공유 백엔드로 우회 |
| NanoClaw와 봇 토큰 충돌 | NanoClaw 중지 후 PoC 실행 |
| 그룹 프라이버시 모드 (일반 텍스트 못 봄) | 인라인 버튼으로 해결 |
| Mock 캘린더 교집합 0개 | referenceSlots로 보장 |
| IPv6 타임아웃 | `https.Agent({ family: 4 })` 적용 |
