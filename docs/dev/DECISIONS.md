## ADR-001: 기존 메신저와 "경쟁"이 아닌 "계층 추가" 포지셔닝

- **날짜**: 2026-03-19
- **결정**: 카카오톡/텔레그램을 대체하지 않고, 그 위에 에이전트 통신 계층을 올린다
- **이유**: 메신저의 네트워크 이펙트는 극복 불가. HTTP가 TCP/IP를 대체하지 않았듯, 새 패러다임은 기존 인프라 위에 올라간다. NanoClaw의 채널 시스템이 이미 이 구조의 프로토타입.
- **대안**: 완전히 새 메신저 (네트워크 이펙트 장벽), 기존 메신저 플러그인 (기능 제한)
- **참고**: `docs/AGENT-MESSENGER-ANALYSIS.md` 라운드 1 합의

## ADR-002: "제품 먼저, 프로토콜은 추출" 전략

- **날짜**: 2026-03-19
- **결정**: 킬러 제품을 먼저 만들고, 내부 통신 규약을 나중에 오픈 스펙(ACP)으로 추출
- **이유**: Matrix(12년, 메인스트림 실패), ActivityPub(Mastodon 1000만 MAU) 등 프로토콜 퍼스트 전략의 한계. MCP는 Claude Code(제품)에서 출발해 프로토콜로 확산됨 — 이 경로를 따른다.
- **대안**: 프로토콜 퍼스트 (채택 느림), 기존 프로토콜 위에 빌드 (제약 많음)
- **참고**: `docs/AGENT-MESSENGER-ANALYSIS.md` 라운드 2 합의

## ADR-003: Developer-first GTM, "에이전트 투명성"이 wedge

- **날짜**: 2026-03-19
- **결정**: 개발자/AI 파워유저를 첫 타깃으로, "내 에이전트가 뭘 하는지 보이게 하는 것"이 핵심 가치 제안
- **이유**: 개발자가 에이전트를 가장 많이 쓰고, 투명성/HITL 니즈가 가장 강함. Stripe/Vercel/Supabase의 developer-first GTM이 증명된 전략.
- **대안**: 일반 사용자 대상 킬러 UX (activation barrier 높음), B2B 엔터프라이즈 (긴 영업 주기)
- **참고**: `docs/AGENT-MESSENGER-ANALYSIS.md` 라운드 3 합의

## ADR-004: PoC는 NanoClaw 통합이 아닌 독립 스크립트

- **날짜**: 2026-03-20
- **결정**: "약속 잡기" PoC를 `poc/schedule-agent/`에 독립 스크립트로 구현
- **이유**: NanoClaw 코드를 수정하면 기존 기능이 깨질 위험. 독립 스크립트로 빠르게 가설 검증 후, 검증된 패턴만 NanoClaw에 통합. 실패해도 NanoClaw에 영향 없음.
- **대안**: NanoClaw IPC 확장 (코드 복잡도 증가, 위험), Telegram Mini App (과도한 복잡도)

## ADR-005: 에이전트 간 통신은 공유 백엔드 경유

- **날짜**: 2026-03-20
- **결정**: Telegram 봇 간 통신을 공유 백엔드(in-memory state machine)로 구현
- **이유**: Telegram은 봇 간 메시지 수신을 원천 차단 (공식 정책). Slack/Discord도 채널 경유만 가능. 어떤 메신저에서든 에이전트 간 직접 통신은 불가능하므로, 별도 통신 계층이 필수. 이것이 ACP의 존재 이유 자체.
- **대안**: Telegram 내 메시지 relay (봇 간 불가), 외부 HTTP API (과도한 인프라)
- **참고**: `docs/research/05-messenger-agent-comparison.md`

## ADR-006: 순차 선택 HITL 패턴 (A 먼저 → B는 A 선택 중에서)

- **날짜**: 2026-03-20
- **결정**: 시간 선택 시 양쪽 동시가 아닌, A가 먼저 복수 선택 → B가 A의 선택지 중에서 선택
- **이유**: 병렬 선택 시 양쪽이 다른 시간을 고르면 교착 상태. 순차 방식은 B의 선택지가 이미 A에 의해 필터링되어 합의 확률이 높음. PoC 테스트에서 자연스러운 플로우 확인.
- **대안**: 병렬 선택 + 교집합 (불일치 시 재투표 필요), 에이전트 자동 결정 (HITL 부재)

## ADR-007: 캘린더 통합은 MCP 서버 추상화 계층으로

- **날짜**: 2026-03-20
- **결정**: Google/Apple/Outlook 캘린더를 MCP 서버로 추상화하여 에이전트는 단일 인터페이스로 접근
- **이유**: 사용자마다 다른 캘린더 사용 (Google 60%, Apple 25%, Outlook 15%). 에이전트는 "빈 시간 알려줘"만 요청하고, 어떤 캘린더인지는 MCP 서버가 라우팅. MCP가 이미 이 패턴을 위해 설계됨. 추가로 "시간 잠금" 등 자체 기능은 래퍼에서 처리.
- **대안**: 각 캘린더 API를 에이전트가 직접 호출 (복잡도 폭발), 자체 캘린더 서비스 (기존 데이터 마이그레이션 문제)
